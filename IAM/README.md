## 2. IAM 4대 구성요소
- ① 사용자(User)   → 사람 1명 (로그인 계정)
- ② 그룹(Group)    → 사람들을 묶는 단위
- ③ 정책(Policy)   → "무엇을 허용한다"는 권한 내용 문서 (부여를 안하면 그냥 문서일 뿐 - 1,2,4 들에게 부여해 줘야함)
- ④ 역할(Role)     → **AWS 서비스**에게 임시로 **권한을 빌려줌** 

### 2-1. 비유로 정리
- 사용자(User) = 직원 개인 출입증 (이름표 있음, 항상 소지)
- 그룹(Group) = 부서 (같은 부서끼리 같은 출입 권한)
- 정책(Policy) = 출입증에 적힌 "어느 방에 들어갈 수 있는지" 규칙
- 역할(Role) = 방문 업체에게 잠깐 빌려주는 임시 출입증 (필요할 때만, 일을 마치면 반납)

### 2-2. Policy 요약
```text
정책(Policy) = 권한 내용 (무엇을 할 수 있나)
       │ 붙인다(attach)
       ▼
┌──────────┬──────────┬──────────┐
│ 사용자    │ 그룹      │ 역할      │
│ (사람1명) │ (사람묶음) │ (서비스용) │
└──────────┴──────────┴──────────┘
       │          │          │
   그 권한을 실제로 행사할 수 있게 됨
```

## 3. 사람(직원)을 관리하는 방법" 으로 정확합니다. 실제 모범 사례
- root는 위험하니 평소엔 안 쓴다 → 맞음
- 그룹에 권한 주고 사용자를 묶는다 → 맞음 (사람 관리할 때)
```text
root 계정 (최고 권한, 평소엔 안 씀)
   │
   └─ IAM 그룹: "개발팀" (권한 설정)
         ├─ IAM 사용자: 김개발
         └─ IAM 사용자: 이코딩
```

## 5. 사용자 종류
- 루트 사용자 : 결제 관리를 포함한 계정의 모든 권한을 가지고 있음
- IAM 사용자 : `IAM`을 통해 생성해서 사용하는 사용자
  - 한 사람 혹은 리소스를 의미
  - 설정 시 콘솔 로그인 권한 부여 가능
  - 설정 시 AWS 서비스를 이용할 수 있음 (지정 권한 내)
      - Access Key
      - Secret Key
  - `AdminAccess` 권한을 부여하더라도 별도의 추가 설정을 하지 않으면 Billing 기능 사용 불가능
