## Q1. 한 회사가 여러 가용성 영역에 배포된 Amazon RDS 인스턴스에서 실행되는 데이터베이스를 호스팅합니다. 주기적 스크립트는 데이터베이스를 쿼리하여 중요한 애플리케이션에 부정적인 영향을 미칩니다. 최소한의 비용으로 애플리케이션 성능을 개선할 수 있는 방법은 무엇입니까?

- B. 개발팀에 하루가 끝나면 수동으로 새 항목을 내보내도록 지시합니다.

- C. Amazon ElastiCache를 사용하여 스크립트가 실행하는 일반적인 쿼리를 캐시합니다.

- D. 데이터베이스의 읽기 복제본을 만듭니다. 읽기 복제본만 쿼리하도록 스크립트를 구성합니다.

# Q3. 한 회사가 온프레미스 데이터 센터 네트워크를 새로운 VPC에 연결해야 합니다. 데이터 센터 네트워크는 100Mbps 대칭형 인터넷 연결을 사용합니다. 온프레미스에서 실행되는 애플리케이션은 매일 수 기가바이트의 데이터를 전송합니다. 이 애플리케이션은 Amazon Data Firehose 전송 스트림을 사용하여 데이터를 처리합니다.
최대 성능을 위해 솔루션 아키텍트는 무엇을 권장해야 합니까?

- A. AWS Snowball Edge Storage Optimized 기기를 구매하세요. 며칠 분량의 데이터가 누적되면 해당 기기에 데이터를 복사하고 AWS로 보내 Firehose로 신속하게 전송하세요.
필요에 따라 반복하세요.

- B. 온프레미스 네트워크와 VPC 간에 VPC 피어링 연결을 생성합니다. 온프레미스 네트워크가 VPC 피어링 연결을 사용하도록 라우팅을 구성합니다.

- C. 온프레미스 네트워크와 VPC 간에 AWS Site-to-Site VPN 연결을 생성합니다. 고객 게이트웨이와 가상 프라이빗 게이트웨이 간에 BGP 라우팅을 구성합니다. VPN 연결을 사용하여 온프레미스에서 Firehose로 데이터를 전송합니다.

- D. AWS PrivateLink를 사용하여 VPC에 Firehose용 인터페이스 VPC 엔드포인트를 생성합니다. 온프레미스 네트워크와 AWS 간에 1Gbps AWS Direct Connect 연결을 설정합니다. PrivateLink 엔드포인트를 사용하여 온프레미스에서 Firehose로 데이터를 전송합니다.

# Q4.한 회사에는 Amazon DynamoDB 테이블에 사용자 거래를 저장하는 웹 애플리케이션이 있습니다. 규정을 준수하기 위해 회사는 사용자 거래 데이터 사본을 7년 동안 보관해야 합니다.
어떤 솔루션이 운영 비용을 최소화하면서 이러한 요구 사항을 충족할 수 있을까요?

- A. DynamoDB를 사용하여 테이블의 온디맨드 백업을 생성합니다. 백업을 Amazon S3 버킷에 저장합니다. S3 버킷에 대한 S3 수명 주기 구성을 설정합니다.
- B. AWS Lambda 함수를 호출하는 Amazon EventBridge 규칙을 생성합니다. Lambda 함수가 테이블을 백업하고 Amazon S3 버킷에 백업을 저장하도록 구성합니다. S3 버킷에 대한 S3 수명 주기 구성을 설정합니다.
- C. AWS Backup을 사용하여 테이블에 대한 백업 일정과 보존 정책을 만듭니다.
- D. DynamoDB 시점 복구를 사용하여 테이블을 지속적으로 백업합니다.

- C. S3 버킷을 만듭니다. S3 버전 관리를 활성화합니다. 보관 기간이 있는 S3 객체 잠금을 사용합니다. CloudFront 배포를 만듭니다. 버킷 정책을 사용하여 액세스를 제한합니다.

# Q6. 한 회사가 여러 개의 프라이빗 서브넷으로 구성된 VPC를 보유하고 있으며, 각 서브넷에는 여러 애플리케이션이 호스팅됩니다. 이러한 애플리케이션은 인터넷에서 접근할 수 없어야 합니다. 하지만 여러 AWS 서비스에 접근해야 합니다. 또한, 애플리케이션은 AWS 서비스에 접근하기 위해 공용 IP 주소를 사용해서는 안 됩니다.


- A. 각 프라이빗 서브넷에 인터넷 게이트웨이를 배포합니다. 프라이빗 서브넷에서 발생하는 트래픽을 인터넷 게이트웨이를 통해 라우팅합니다.

- B. 프라이빗 서브넷 간에 AWS Direct Connect 연결을 설정합니다. 프라이빗 서브넷에서 Direct Connect 연결을 통해 트래픽을 라우팅합니다.

- C. 필요한 AWS 서비스에 대한 인터페이스 VPC 엔드포인트를 구성합니다. 프라이빗 서브넷에서 인터페이스 VPC 엔드포인트를 통해 트래픽을 라우팅합니다.

- D. 각 프라이빗 서브넷에 NAT 게이트웨이를 배포합니다. 프라이빗 서브넷에서 발생하는 트래픽을 NAT 게이트웨이를 통해 라우팅합니다.

# Q7. 한 회사가 Amazon EC2 인스턴스를 사용하여 퍼블릭 웹 애플리케이션에 대한 요청을 처리합니다. 해당 애플리케이션은 URL 경로를 사용하여 트래픽을 여러 애플리케이션 페이지로 라우팅합니다.
회사는 예측할 수 없는 시간에 엄청난 트래픽 급증을 경험하기 시작했습니다. 트래픽 급증으로 인해 웹 애플리케이션에 문제가 발생하고 가끔씩 서비스를 이용할 수 없게 되었습니다.
회사는 트래픽이 갑자기 늘어나는 것을 처리하기 위해 웹 애플리케이션의 확장성을 높여야 합니다.
어떤 솔루션이 이 요구 사항을 충족시킬까요?


- A. 애플리케이션 로드 밸런서(ALB)를 생성합니다. 웹 애플리케이션 인스턴스를 대상으로 설정합니다. ALB 트래픽 지표를 기반으로 Amazon CloudWatch 알람을 생성합니다. 트래픽 급증 시 활성화되도록 알림을 구성합니다.

- B. 웹 애플리케이션 인스턴스의 Amazon Machine Image(AMI)를 생성합니다. AMI를 사용하여 별도의 가용 영역에 두 개의 웹 애플리케이션 인스턴스를 더 생성합니다. 세 인스턴스를 모두 참조하도록 웹사이트 DNS 레코드를 업데이트합니다.

- C. 애플리케이션의 Docker 이미지를 생성합니다. Amazon Elastic Container Service(Amazon ECS)를 사용하여 Auto Scaling ECS ​​클러스터를 생성합니다. 관리형 확장을 활성화합니다. 네트워크 로드 밸런서를 생성합니다. ECS 클러스터를 대상 그룹으로 설정합니다.

- D. 웹 애플리케이션 인스턴스의 Amazon Machine Image(AMI)를 생성합니다. AMI를 사용하여 최소 용량이 2개인 EC2 인스턴스로 구성된 Auto Scaling 그룹을 생성합니다. 애플리케이션 로드 밸런서를 생성합니다. Auto Scaling 그룹을 대상 그룹으로 설정합니다.

- A. 매달 AWS Cost Explorer를 사용하여 SFTP 전송 비용을 검토하고 고객별 세부 내역을 확인하세요. 해당 요금을 고객의 월별 청구서에 추가하세요.

- B. 소프트웨어를 호스팅하는 S3 버킷에서 요청자 결제를 활성화합니다. 고객의 요청에 따라 각 고객에게 요금을 할당합니다.

- C. 매달 로깅 S3 버킷에 Amazon Athena 쿼리를 실행하여 고객 사용량을 파악하고 비용을 계산합니다. 해당 요금을 고객의 월별 청구서에 추가합니다.

- D. VPC 흐름 로그를 새 S3 버킷에 기록하도록 구성합니다. 흐름 로그에 대해 매월 쿼리를 실행하여 고객 사용량을 파악하고 비용을 계산합니다. 해당 요금을 고객의 월별 청구서에 추가합니다.

# Q49. 한 회사에서 JSON 웹 토큰(JWT)을 사용하여 사용자 권한을 부여하는 API를 만들고자 합니다. 또한 경로 기반 라우팅을 사용하여 여러 AWS 서비스에 대한 동적 액세스를 지원해야 합니다.
어떤 솔루션이 이러한 요구 사항을 충족시킬까요?

- A. Amazon API Gateway HTTP API 뒤에 네트워크 로드 밸런서를 배포합니다. 권한 부여에는 Amazon Cognito를 사용하세요.

- B. Amazon API Gateway HTTP API 뒤에 애플리케이션 로드 밸런서를 배포합니다. 권한 부여에는 Amazon Cognito를 사용합니다.

- C. Amazon API Gateway REST API 뒤에 네트워크 로드 밸런서를 배포합니다. AWS Lambda 함수를 사용자 지정 권한 부여자로 사용합니다.

- D. Amazon API Gateway REST API 뒤에 애플리케이션 로드 밸런서를 배포합니다. IAM 권한 부여를 구성합니다.

# Q51. 한 회사가 민감한 고객 데이터를 처리하는 AWS 기반 클라우드 기반 애플리케이션을 구축하고 있습니다. 이 애플리케이션은 데이터베이스로 Amazon RDS를, 객체 스토리지로 Amazon S3를, 그리고 서버리스 처리를 위해 AWS Lambda를 호출하는 S3 이벤트 알림을 사용합니다.
이 회사는 AWS IAM Identity Center를 사용하여 사용자 자격 증명을 관리합니다. 개발, 테스트 및 운영 팀은 민감한 고객 데이터의 기밀성을 보장하는 동시에 Amazon RDS 및 Amazon S3에 안전하게 액세스할 수 있어야 합니다. 솔루션은 최소 권한 원칙을 준수해야 합니다.
어떤 솔루션이 운영 오버헤드를 최소화하면서 이러한 요구 사항을 충족시킬 수 있을까요?


- A. 모든 팀의 각 구성원에게 역할 기반 권한을 부여하는 개별 IAM 사용자를 생성합니다. 사용자 요구에 따라 RDS 및 S3 접근 권한에 대한 사전 정의된 정책을 사용하여 각 사용자에게 IAM 역할을 할당합니다. 정기적인 자격 증명 평가를 위해 IAM Access Analyzer를 구현합니다.

- B. AWS Organizations를 사용하여 각 팀에 대해 별도의 계정을 생성하세요. 최소 권한으로 교차 계정 IAM 역할을 구현하세요. 팀 역할 및 책임에 따라 RDS 및 S3 액세스에 대한 특정 권한을 부여하세요.

- C. 모든 팀에 최소 권한의 IAM 역할을 사용하여 액세스 권한을 부여합니다. 팀의 책임에 따라 Amazon RDS 및 S3 객체 액세스에 대한 특정 권한을 정의하는 맞춤형 IAM 정책을 사용하여 각 팀에 IAM 역할을 할당합니다.

- D. Identity Center 디렉터리를 사용하여 IAM Identity Center를 활성화합니다. Amazon RDS 및 Amazon S3에 대한 세부적인 접근 권한을 가진 권한 집합을 생성하고 구성합니다. 모든 팀을 해당 권한 집합을 통해 특정 접근 권한을 가진 그룹에 할당합니다.
